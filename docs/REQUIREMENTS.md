# 要件定義書 — Jizake（日本酒レコメンドアプリ）

> 本書はユーザーから提示された要望を要件として整理したもの。
> 自律実行モードのため、曖昧な点は推奨デフォルトで補い「前提・仮定」に明記した。

## 1. プロジェクト概要

日本酒のデータベースを構築し、ユーザーに対して以下を提供する Web アプリケーション。

- おすすめの日本酒の表示（閲覧・検索履歴ベースのパーソナライズ推薦）
- 都道府県を選んで地酒を表示
- 名前・都道府県・味による検索と検索結果一覧
- 各日本酒の詳細ページ（説明・タグ・外部リンク・価格帯）
- チャット Q&A 形式で欲求をヒアリングし、RAG を用いて日本酒を提案するチャットボット

## 2. 機能要件

### FR-01: 日本酒データの取得・管理
- 日本酒データを外部ソースから取得、またはシードデータとして投入し、DB で管理する。
- 各日本酒は最低限、名称・蔵元・製造都道府県・説明文を持つ。

**受け入れ条件**
- [ ] 日本酒データが DB に格納され、一覧・詳細 API で取得できる。
- [ ] データ投入（シード/インポート）が再実行可能な手順として整備されている。

### FR-02: 特徴タグ
- 各日本酒に特徴タグ（製造都道府県・味わい〔甘口/辛口/淡麗/濃醇 等〕・種別〔純米/吟醸/大吟醸 等〕など）を付与できる。
- タグは多対多で管理し、タグからの逆引き（タグ→日本酒一覧）ができる。

**受け入れ条件**
- [ ] 日本酒詳細でタグ一覧が表示される。
- [ ] タグをキーに日本酒を絞り込める。

### FR-03: 日本酒詳細ページ
- 各日本酒に説明をまとめた詳細ページを持つ。
- 表示項目: 説明、タグ、公式紹介ページリンク、Amazon 等の購入リンク、価格帯（取得できる場合）。

**受け入れ条件**
- [ ] `/sake/[id]` 形式の URL で詳細ページへ直接アクセスできる。
- [ ] 外部リンクは別タブで開き、リンクが無い場合は非表示になる。

### FR-04: ログイン機能
- ユーザー登録・ログイン・ログアウトができる。
- 履歴・推薦などパーソナライズ機能はログインユーザーに紐づく。
- 未ログインでも閲覧・検索は可能（推薦・履歴のみログイン必須）。

**受け入れ条件**
- [ ] メール等でサインアップ／ログイン／ログアウトできる。
- [ ] 未ログインで履歴・パーソナライズ推薦にアクセスすると誘導される。

### FR-05: 履歴ベースの推薦
- ユーザーの閲覧履歴・検索履歴を記録する。
- 履歴から嗜好（よく見るタグ・都道府県・味）を推定し、おすすめの日本酒を表示する。

**受け入れ条件**
- [ ] 詳細ページ閲覧と検索実行が履歴として記録される。
- [ ] ホーム画面に履歴に基づくおすすめが表示される（履歴が無い場合は人気・ランダム等のフォールバック）。

### FR-06: 検索機能
- 名前（部分一致）・都道府県・味（タグ）で検索できる。複合条件も可能。
- 検索結果は一覧表示し、一覧から選択すると詳細ページへ遷移する。

**受け入れ条件**
- [ ] 名前・都道府県・味の各条件および組み合わせで検索できる。
- [ ] 結果一覧のカードから詳細ページに遷移できる。
- [ ] 該当 0 件の場合は空状態メッセージが表示される。

### FR-07: 都道府県から地酒を表示
- 都道府県を選択すると、その県の地酒一覧が表示される。

**受け入れ条件**
- [ ] 都道府県の選択 UI（マップまたはリスト）から一覧に到達できる。

### FR-08: RAG チャットボット
- 「どんなお酒を求めていますか？」から始まる Q&A 形式で数問ヒアリングし、回答に応じた日本酒を複数提案する。
- 回答生成には RAG（日本酒 DB を知識源とした検索拡張生成）を用い、酒に特化した回答を行う。
- 提案された日本酒から詳細ページへ遷移できる。

**受け入れ条件**
- [ ] チャット UI で質問→回答のやり取りができ、最終的に日本酒が複数提案される。
- [ ] 提案はアプリ内 DB に存在する日本酒であり、詳細ページへのリンクを持つ。
- [ ] DB に無い銘柄を捏造して提案しない（RAG の検索結果に基づく）。

### FR-09: 銘柄画像の表示（ベストエフォート）
- 各日本酒のパッケージ（ボトル）画像を、一覧カード・詳細ページに表示する。
- 画像は楽天市場 商品検索 API から取得した商品画像 URL（楽天 CDN）を参照する。
  取得できない銘柄は共通の「No Image」プレースホルダを表示する（実在商品の誤った画像を
  出すくらいなら出さない、は維持。FR-08 の捏造防止と同じ思想。プレースホルダは
  商品写真と誤認されない意匠とする）。
- 画像取得と同時に楽天商品ページ URL（購入リンク）も取得し、`rakuten_url` の欠損を補完する。

**受け入れ条件**
- [ ] 画像取得バッチが再実行可能な手順として整備されている（冪等・差分実行）。
- [ ] 画像がある銘柄はカード・詳細ページに画像が表示され、無い銘柄は共通の No Image プレースホルダが表示される。
- [ ] 誤マッチ抑止（銘柄名の包含・セット商品等の除外）を通過した画像のみ表示し、
      マッチ結果を目視監査できる出力（ログ/CSV）がある。
- [ ] 詳細ページに画像の出典（楽天市場）と商品ページへの導線が表示される。

## 3. 非機能要件

| 区分 | 要件 |
|---|---|
| 性能 | 検索・一覧表示は通常操作で 2 秒以内に表示。チャット応答はストリーミング表示。 |
| セキュリティ | パスワード等の資格情報は適切にハッシュ化。シークレットはコミットしない。履歴は本人のみ参照可能。 |
| 可用性 | 個人開発規模。マネージドサービス利用で運用負荷を最小化。 |
| コスト | 無料枠〜低コストで運用できる構成を優先。 |
| 対応環境 | モバイル含むレスポンシブ Web。日本語 UI。 |

## 4. スコープ外（当面）

- EC 機能（購入はあくまで外部リンク誘導）
- レビュー・コメント等の UGC 投稿機能
- ネイティブアプリ
- 管理画面（初期はシードスクリプトでデータ管理）

## 5. 前提・仮定

- 日本酒のオープンな公式 API は限られるため、データ取得は「公開データセット／自作シードデータ＋段階的な拡充」を基本とする（詳細は FEASIBILITY で調査）。
- 価格帯・Amazon リンクは取得できる範囲で表示するベストエフォート項目とする。
- 銘柄画像は楽天市場 API で取得できる範囲のベストエフォート項目とする（FR-09。
  画像は楽天 CDN の URL を参照表示し、自前でダウンロード・加工・保存はしない）。
- 推薦は初期はルールベース（タグ頻度ベース）で十分とし、将来の協調フィルタリング等は拡張余地として設計する。
