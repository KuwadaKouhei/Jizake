# 実装思想（CODING_PHILOSOPHY）

> 自律実行モードで作成したため、推奨デフォルトを採用。異論があれば更新して以降を追従させる。

## 原則

1. **フレームワーク規約に従う（Convention over Configuration）**
   採用フレームワークの標準的な書き方・配置を最優先する。独自レイヤーや独自規約を発明しない。

2. **型は境界で厳格に**
   TypeScript `strict: true`。API 入出力・DB スキーマ・LLM 応答など「境界」では必ず型（と実行時バリデーション）を定義する。
   内部の細かい型注釈は推論に任せてよい。

3. **早すぎる抽象化をしない（Rule of Three）**
   同じコードが 3 回現れるまで共通化しない。`utils/common` の肥大化を禁止する。

4. **コロケーション**
   その機能でしか使わないコンポーネント・ロジック・型は、その機能のディレクトリに置く。
   横断利用が確定したものだけを共有領域へ昇格させる。

5. **エラーは境界で捕まえ、ユーザーには回復手段を示す**
   - 外部 API（LLM・データソース）呼び出しは失敗を前提に書く（タイムアウト・フォールバック文言）。
   - 想定内エラーは Result 的に扱い、想定外は上位のエラーバウンダリに任せる。

6. **小さくコミット、常に動く main**
   縦スライス（画面〜DB を貫く最小機能）単位で実装し、各コミットでアプリが起動する状態を保つ。

## スタイル

- 命名: 変数・関数は camelCase、型・コンポーネントは PascalCase、ファイルは kebab-case（コンポーネントは PascalCase 可。フレームワーク規約優先）。
- コメントは「コードから読み取れない制約・理由」のみ書く。処理の逐語訳コメントは書かない。
- マジックナンバー・文言は定数化する（特に推薦スコアの重み・RAG のプロンプト）。
- Lint/Format はツール（ESLint + Prettier 等）に完全委任し、人間はスタイル議論をしない。

## 秘密情報

- API キー等は環境変数でのみ扱い、`.env*` はコミットしない。クライアントコードにシークレットを埋め込まない。
