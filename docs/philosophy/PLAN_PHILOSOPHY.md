# 設計思想（PLAN_PHILOSOPHY）

> 自律実行モードで作成したため、インタビューの代わりに推奨デフォルトを採用している。
> 方針に異論がある場合はこのファイルを更新し、以降の設計・実装を追従させること。

## コア原則（優先順位順）

1. **シンプルさ最優先（Simple over Clever）**
   個人開発規模のアプリである。凝った抽象化・先回りの汎用化より、読んで追える素直な構造を選ぶ。
   迷ったら「機能が少なく単純な方」を採用する。

2. **データ中心設計（Data-First）**
   このアプリの価値の源泉は「日本酒データとタグ」である。
   スキーマ（Sake / Tag / History）を設計の中心に置き、検索・推薦・RAG はすべて
   「同じ日本酒データを異なる方法で引く」機能として統一的に捉える。

3. **差し替え可能な知能（Replaceable Intelligence）**
   推薦ロジックと RAG は今後最も変化しやすい部分である。
   - 推薦: 初期はルールベース（タグ頻度スコア）。インターフェース（入力=ユーザー履歴、出力=日本酒リスト）を固定し、実装を差し替え可能にする。
   - RAG: 検索部（retriever）と生成部（LLM）を分離し、埋め込みモデルや LLM の変更がアプリ本体に波及しないようにする。

4. **マネージド優先・運用ゼロ志向**
   自前でサーバーやミドルウェアを運用しない。DB・認証・ホスティング・LLM はマネージドサービスを使い、
   無料枠〜低コストに収める。

5. **未ログインでも価値がある（Progressive Personalization)**
   閲覧・検索・チャットは未ログインで使え、ログインすると履歴・パーソナライズが加わる。
   認証を機能のゲートにしない。

## 境界と依存方向

- UI → アプリケーションサービス → データアクセス の一方向依存。逆流禁止。
- 外部サービス（LLM、データソース、認証）はアダプタ越しに使い、ベンダー型をドメイン層へ漏らさない。
- 推薦・RAG・検索は「日本酒カタログ（DB）」に依存するが、互いには依存しない。

## トレードオフの決め方

| 対立 | 選ぶ側 | 理由 |
|---|---|---|
| 開発速度 vs 完成度 | 開発速度 | 縦スライスで動くものを早く出し、履歴データを貯め始める方が価値が大きい |
| 正規化 vs 読み取り性能 | 正規化 | データ件数は高々数千件。JOIN コストより整合性を優先 |
| 自前実装 vs 外部サービス | 外部サービス | 運用ゼロ志向。ただしロックインはアダプタで緩和 |
| リアルタイム性 vs コスト | コスト | 推薦の再計算はオンデマンドで十分 |

## 逸脱のルール

思想から逸脱する設計判断をする場合は、該当ドキュメント（DESIGN.md 等）に「逸脱と理由」を必ず残す。
