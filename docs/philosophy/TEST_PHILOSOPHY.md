# テスト思想（TEST_PHILOSOPHY）

> 自律実行モードで作成したため、推奨デフォルトを採用。異論があれば更新して以降を追従させる。

## 基本方針

**「壊れると一番困るのは、データを引くロジックと推薦・検索の正しさ」**。
UI の見た目より、ドメインロジック（検索条件の組み立て・推薦スコアリング・RAG の検索部）を厚くテストする。

## テスト構成（目標比率）

| 種別 | 対象 | 比率目安 | ツール例 |
|---|---|---|---|
| ユニット | 推薦スコアリング、検索クエリ組み立て、タグ処理、ユーティリティ | 60% | Vitest |
| 統合 | API ルート／サーバーアクション（テスト用 DB 使用）、RAG リトリーバ | 30% | Vitest + テストDB |
| E2E | 主要動線のみ: 検索→一覧→詳細、ログイン、チャット1往復 | 10% | Playwright |

## モック方針

- **LLM API は必ずモックする**（コスト・非決定性のため）。RAG は「リトリーバが正しい文書を返すか」を実 DB でテストし、生成部は固定応答でテストする。
- 自作モジュール同士のモックは最小限にし、なるべく本物を組み合わせて統合テストする。
- 外部データソース（さけのわ API 等）はフィクスチャ（保存済みレスポンス）でテストする。

## 受け入れ条件との対応

- REQUIREMENTS.md の各受け入れ条件に対応するテスト（またはE2E）を最低 1 つ用意する。
- バグ修正時は、再現テストを先に書いてから直す。

## CI ゲート

- PR で lint / typecheck / unit+integration を必須にする。E2E は主要動線のみ（時間コストとのバランス）。
